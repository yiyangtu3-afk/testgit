# CampusLink Vue 前端渐进迁移交接

本文定义 CampusLink 从原生 ES Modules 静态前端迁移到 Vue 3 的方式。迁移从
独立目录开始，保留当前可演示版本，避免在尚未验证等价行为前替换入口。

## 当前状态

当前前端不是 Vue、React 或其他 SPA 框架，而是根目录 `index.html`、`app.js`、
`styles.css` 与 `frontend/js/` 中按职责拆分的原生 ES Modules。它已有 API、认证、
聊天、动态、活动、通知和管理员模块；`frontend/js/state.js` 是所有旧模块共享的
状态单例。当前可用版本和业务基线见
[`new-chat-handoff-2026-07-08.md`](new-chat-handoff-2026-07-08.md)。

第一切片已经完成：`frontend-vue/` 现在包含独立的 Vue 3、Vite、Vue Router 和
Pinia 应用，以及最小认证闭环。根目录静态入口和 `frontend/js/` 仍是功能基线和
默认演示入口；不要删除、移动、替换或重写它们。

第二切片已经完成：Vue 应用壳提供导航、当前账号和统一的 Java API/Mock 状态提示。
它不读取聊天、动态、活动、通知或管理员业务数据；这些领域继续只在旧版可用。

第三切片已经完成：Vue 提供联系人、好友申请和好友会话。它复用现有 JWT API、游标
分页、附件、未读数和 `/ws/chat` 实时通道；动态、活动、通知和管理员领域仍未迁移。

第四切片已经完成：Vue 提供公开动态、个人动态、点赞、评论和审核状态反馈。公开流只
显示已审核内容，个人动态清楚显示待审、通过或拒绝原因；活动、通知和管理员仍未迁移。

第五切片已经完成：Vue 提供活动筛选、报名、候补、取消、组织者名单和签到。学生不会
请求组织者专属运营接口，因此 Java API 的权限边界保持可见。

第六切片已经完成：Vue 的统一通知中心按时间合并活动和社交通知，支持单条或全部已读、
活动/动态/好友申请的受限目标跳转，并复用已认证的 `/ws/chat` 连接接收实时事件。
通知 store 按 ID 去重；Java API 的 HTTP 失败仍保留真实错误，不会回退 Mock。

第七切片已经完成：Vue 管理控制台在管理员身份下读取指标、待审核活动、待审核内容、
审计记录和报表。组件不会为非管理员发起后台请求；所有操作继续经过既有 JWT API、
Mock 边界和后端角色校验。

## 目标架构

新前端使用 Vue 3、Vite、Vue Router 和 Pinia，初始放在仓库根目录的
`frontend-vue/`。选择独立目录可以让旧版继续在 `5179` 演示，并让新旧版本在功能
完全等价前并行验证。

建议的初始结构如下：

```text
frontend-vue/
├── package.json
├── vite.config.js
└── src/
    ├── main.js
    ├── App.vue
    ├── router/
    ├── stores/
    ├── services/
    │   ├── api/
    │   └── realtime/
    ├── features/
    │   ├── auth/
    │   ├── chat/
    │   ├── feed/
    │   ├── activities/
    │   ├── notifications/
    │   └── admin/
    └── shared/
```

`services/api/` 必须成为新应用唯一的 HTTP 边界；`stores/` 只保存跨页面状态；
`features/` 只组合本领域组件、事件与加载流程。不要把旧的单一 `state.js` 原样复制到
Pinia，也不要让组件直接散落 `fetch` 调用。

## 不可改变的行为

迁移只替换前端表现和组织方式，不能改变已验证的后端、权限或数据语义。

- Java API 可达时，显示本地 MySQL 历史数据；仅在网络层完全不可达时才可使用 Mock。
  API 返回 `4xx` 或 `5xx` 时必须展示真实失败，绝不能写入或回退 Mock。
- 所有受保护请求继续发送当前 bearer token。前端不得提交或信任当前用户、管理员、
  申请人或收件人的身份字段来代替服务端 JWT 身份。
- 保留现有 WebSocket 聊天和社交通知的收件人定向、断线处理和历史加载语义。
- 保留 XSS 转义、好友聊天边界、通知目标受限解析、审核状态和管理员角色限制。
- `frontend/js/state.js` 的旧版导入不能添加版本查询参数。迁移期间旧代码仍要共享同一
  状态单例。
- 不重置、重种、清空或清理本地 MySQL 历史数据；不运行 `git reset --hard`、
  `git clean`，也不删除未跟踪文件。

## 第一阶段：脚手架、认证与 API 边界（已完成）

第一阶段只建立 Vue 工程和最小可验证闭环，不替换根目录 `index.html`，不迁移聊天、
动态、活动、通知或管理员页面。

1. 在 `frontend-vue/` 初始化 Vue 3、Vite、Vue Router 和 Pinia，同时保留旧前端的
   所有文件和运行脚本。
2. 将 Vite 固定在 `127.0.0.1:5180`，并把 `/api` 和 `/ws` 代理到
   `http://127.0.0.1:8080`。所有新请求使用相对路径，因此没有修改后端 CORS。
3. 将 HTTP 调用集中到 `services/api/`。`ApiUnavailableError` 只表示网络层不可达；
   `ApiHttpError` 保留 API 的 `4xx` 和 `5xx` 失败，绝不回退 Mock。
4. 新增 Pinia 会话 store，持久化 token 与用户信息，并提供验证码登录、演示登录、
   注销和 Java API/Mock 状态反馈。受保护请求通过 bearer token 注入认证头。
5. 新增 API 边界和会话 store 单元测试。测试覆盖 Mock 登录、token 注入，以及
   `401`、`403`、`500` 不触发 Mock。
6. `npm test`、`npm run build` 和旧版 `./script/run_frontend_check.sh` 已通过。
   通过 Vite 代理验证了本地 Java API 的验证码登录、演示登录和注销，均返回 `200`。
   本阶段未重置、重种或清理本机 MySQL 历史数据。

4. 为 shell Pinia store 添加单元测试。新 Vue 测试、生产构建与旧版前端检查均通过。

## 第六阶段：站内通知与实时事件（已完成）

此阶段迁移现有通知 API 的前端适配，不修改控制器、WebSocket 协议或 MySQL 数据。

1. `notification-api.js` 通过相对 `/api` 路径读取活动和社交通知摘要，并复用统一的
   `ApiUnavailableError` 降级规则。
2. `notifications` Pinia store 按时间合并两类摘要；单条、全部已读和实时事件都更新同一

3. 站内通知页面通过既有受限目标 API 打开动态或好友申请；活动通知直接进入活动页。
4. `AppShell` 保持唯一的已认证 `/ws/chat` 客户端，同时分发聊天事件和通知事件，聊天
   页面不会再创建第二条连接。
5. 通知 API 边界测试明确覆盖 `401`、`403`、`500` 不回退 Mock；通知 store 测试覆盖
   合并、已读和实时去重。`npm test`、`npm run build` 与旧版前端检查均通过。

本机 8080 服务在本次前端验证时未运行，因此没有把不可达状态伪装成 live 验收，也没有

## 第七阶段：管理员控制台（已完成）

此阶段复用现有管理员端点，不新增后端 API、CORS 规则或数据库写入逻辑。

1. `admin-api.js` 覆盖指标、活动审核、内容审核、审计记录、批量删除和报表端点。
2. `admin` Pinia store 在组件加载前确认管理员角色；非管理员只看到权限说明，且不会
   请求管理员 API。
3. 管理界面提供活动同意/拒绝、内容同意/拒绝、审核及审计记录批量删除、今日/本周/全部
   报表和 CSV 下载。内容拒绝必须填写审核意见；内容审核工作台只请求和显示待审核内容，
   并支持按类型和提交人组合筛选。已完成决定从该操作区移除，审核详情仍保留在持久化数据
   和审计记录中。
4. API 边界测试覆盖 `401`、`403`、`500` 不回退 Mock；store 测试覆盖管理员加载、审核
   刷新和非管理员不请求数据。`npm test`、`npm run build` 和旧版前端检查均通过。

管理员还可为待审核内容主动生成审核辅助建议。当前建议来自本地可解释规则，返回风险等级、
命中信号和建议意见；不会自动审核、预填审核意见或写入审计记录。该调用沿用同一 API/Mock
边界，HTTP `4xx`、`5xx` 仍显示真实错误。

Vue 领域迁移现在完成七个切片。下一步是逐项新旧等价验收；在用户明确授权前，不切换
根目录默认入口，也不删除或替换旧版文件。

## 等价验收记录

2026 年 7 月 17 日开始 live 验收。Java API 与新旧本地预览都可访问；学生读取动态、
活动、好友和两类通知均返回 `200`，访问管理员指标返回 `403`。管理员读取指标、活动
审核、内容审核、审计和报表均返回 `200`。同一 bearer token 下，Vue 代理和旧版直连
API 对动态、活动、通知、指标、审核与审计响应完全一致。

本轮修复 Vue 动态评论面板：评论加载数据原本写入 Pinia store，但展开判断错误地读取
本地草稿状态，导致评论区始终不显示。现在展开状态与草稿分别管理。通过 Vue 代理提交
带日期标识的评论后，该评论进入管理员待审队列；管理员审核通过后，公开评论列表返回
该评论。该验证保留正常的 MySQL 评论与审计历史，没有重置、重种或清理数据。

验收同时发现 Vue 活动页缺少旧版已有的教师/社团负责人创建入口。该入口现已补齐，表单
包含标题、类别、名额、地点、起止时间和详情，不允许客户端填写组织者。store 在提交前
验证结束时间晚于开始时间，成功后明确提示活动处于待审核状态。live 验收通过 Vue 代理
创建带唯一标识的活动，由管理员发布后确认学生可以报名、收到报名通知并取消报名；活动、
审核、报名、通知和审计历史按验证需要保留。

聊天 live 验收通过 Vue 代理发送带唯一标识的消息，确认接收方未读数从 `0` 变为 `1`，
打开会话后清零，发送方撤回后接收方读取到撤回状态。使用接收方 bearer token 建立的
`/ws/chat` 连接还收到 `message.created` 和 `message.withdrawn` 事件。测试消息在验收
期间已撤回；消息与审计历史按现有业务规则保留。

Vue 聊天面板现在使用固定高度的三段布局：联系人列表和对话标题不随消息历史延长，消息流
在面板内部上下滚动，发送输入框与待发送附件区固定在面板底部。窄屏下联系人区仍可先滚动，
选定对话后的消息流和输入框保持独立可用。

对话身份栏与发送区使用紧凑留白，默认输入框显示两行，附件和发送按钮维持可访问的点击
区域但不占用过多高度。输入框仍可拖拽增高，较长草稿不会牺牲可编辑空间。

选择会话或发送消息后，Vue 会将消息流定位到最新内容；加载更早的分页消息会补偿新增高度，
保留用户正在阅读的位置。窄屏提供联系人栏开关，选定会话后可以收起联系人栏并把宽度留给
对话区。

宽屏聊天不再受固定工作区最大宽度限制。应用侧栏、页面外边距和联系人栏采用紧凑宽度，
将剩余空间优先分配给对话区；窄屏仍保持单列布局和联系人栏开关。

聊天页不再重复显示“联系人与对话”和连接状态标题。对话面板直接从工作台页头下开始，尽量
填满可用视口高度；窄屏联系人开关移到对话标题栏，不牺牲消息历史的可视空间。

工作台不再在每个路由上重复显示“认证会话已就绪”状态横幅。登录来源仍保存在会话状态中，
各领域页面继续在自己的操作区显示成功、失败和权限反馈，从而把可视空间留给当前内容。

侧边栏仅保留功能导航名称和当前页面高亮，不再展示迁移进度、完成计数或导航项的解释性
副标题。页面不会用开发迁移信息挤占用户的工作区域。

通知 live 验收标记本轮活动报名通知为已读；通过一次点赞并取消点赞生成社交通知，确认
收件方收到 `social.notification.created`、通知持久化、受限动态目标解析和单条已读均正常。
`markAll` store 测试覆盖两类通知摘要同时更新，不对本机历史通知执行全部已读操作。

管理员 live 验收通过 Vue 代理创建带唯一标识的待审活动，再由管理员拒绝并确认组织者端
保留拒绝原因；创建待审动态并审核通过后，学生公开动态流可读取该内容；为该动态创建待审
评论后，管理员从内容审核队列删除该条记录。验收还删除了本轮新生成的一条审计记录，并确认
`today` 报表返回匹配范围、CSV 文件名和指标。除上述正常业务删除外，验证没有重置、重种或
清理本机 MySQL 历史数据。

注册 live 验收通过 Vue 代理请求验证码、注册带唯一标识的学生账号、以同一手机号再次注册
得到 `409`，并注销临时 JWT。只读 MySQL 查询确认该新用户记录存在；注册用户与审计历史按
真实业务结果保留，没有重置、重种或清理数据。

注册页随后确认 `createApi()` 合并认证与活动适配器时，活动的 `register(id)` 会覆盖认证的
`register(name, phone, code)`。因此学生注册会错误地请求受保护的活动报名接口，并在浏览器留有
失效 token 时显示 `401`“请先登录”。认证动作现在使用明确的 `registerStudent` 名称；活动报名
继续使用 `register`。组合 API 回归测试同时验证学生注册不发送 bearer token，活动报名仍发送
当前 token。

不透明凭证，组织者在运营区输入凭证核验。凭证仅以 SHA-256 摘要持久化，重新展示会轮换旧码；
核验继续由后端统一检查 JWT、组织者归属、活动与报名状态，并复用既有事务签到。该功能没有
回迁到旧版静态入口，Mock 只在网络不可达时提供同形返回。Vue 名单的手动签到也已改为使用
后端实际返回的 `registrationId`。本轮 Vue 单元测试 43 项（16 个文件）和生产构建通过，
并保留 HTTP `4xx`、`5xx` 不回退 Mock 的既有边界。

内容审核现在为每次决定保存 `reviewerName`、`reviewedAt` 和 `reviewComment`。
`POST /api/admin/moderation/{itemId}/{decision}` 接受可选的 `comment` 请求字段，但
`reject` 必须提供非空审核意见；默认的队列查询只返回 `pending`，Vue 也只显示该队列。
架构脚本使用 `information_schema` 条件迁移，
只在现有 MySQL 缺少字段时添加列，不重置或重种历史数据。审核审计事件同时写入审核人、
时间和意见；MySQL 回滚集成测试覆盖这三个字段的持久化。

近期 Vue 体验调整包括：

- 工作台账号菜单可直接切换学生、教师、社团负责人（王社长）和管理员演示身份。
- 学生自助注册使用 `registerStudent` API 动作；在 Java API 可达时注册会持久化到 MySQL，
  API 返回 `401`、`403`、`409` 或 `500` 时保持真实失败，不回退 Mock。
- 聊天页取消重复标题、认证状态横幅和侧栏迁移说明。宽屏优先把空间分给对话区，联系人栏
  保持紧凑宽度。
- 对话使用固定高度的三段布局：标题和输入区固定，只有消息历史内部滚动。最新浏览器验收在
  `1440 × 900` 视口得到约 `500px` 的消息流可视高度；长历史保持内部滚动，输入框仍可见。
- 窄屏联系人开关位于对话标题栏；选择联系人后可收起联系人列，不牺牲消息历史高度。
- 组织者进入 **校园活动** 后，标题区直接显示 **现场签到** 入口。点击后页面定位并聚焦到
  **我的活动运营** 的第一个凭证核验框；学生身份不显示该入口。即使组织者当前没有自己创建的
  活动，入口仍会保留，并说明必须切换到该活动创建者账号才能核验凭证。
- 活动页只为学生读取“我的报名”，避免组织者和管理员触发学生专属接口而中断其运营区加载。每张
  活动卡同时显示 **发起人**，方便参与者切换到正确的教师或社团负责人账号完成现场签到。
- 只有学生看到 **立即报名**。教师、社团负责人和管理员仍可浏览活动，但活动卡显示
  **仅学生可报名**，不会提供无效的报名动作。已完成签到的学生显示 **已签到**，也不会再次
  显示报名入口。
- 只有已发布或满额活动在组织者运营区显示名单、凭证输入和手动签到。待审核或未通过活动保留
  明确状态说明，不能从前端或 API 发起签到。活动加载、报名、取消、名单和手动签到失败时，页面
  显示真实 API 错误，不会静默失败或回退 Mock。

聊天附件的浏览器 UUID 兼容性也已修复：浏览器生成 36 位 UUID，后端
